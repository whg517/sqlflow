# SQLFlow 需求管理规范

> 版本: v1 | 日期: 2026-05-02 | 状态: 当前有效

---

## 1. 目的

本规范定义 SQLFlow 项目中需求的**全生命周期管理方式**，确保：

- 需求从提出到交付全程可追溯
- 文档是驱动开发的唯一真相来源
- 变更有序可控，不因口头沟通导致代码和文档脱节

## 2. 文档体系

### 2.1 规格文档（Spec）

系统"应该是什么样"的唯一定义。存放在 `docs/spec/`，是所有开发的依据。

| 文档 | 职责 | 面向 |
|------|------|------|
| `PRD.md` | 产品规格——做什么、为什么做、给谁用 | 产品、开发 |
| `ARCHITECTURE.md` | 技术架构——怎么构建、用什么技术、如何组织 | 开发 |
| `UI-DESIGN.md` | UI 规格——界面长什么样、怎么交互、视觉规范 | 前端开发 |

**规则**：
- Spec 是真相来源，代码必须与 Spec 保持一致
- Spec 只通过**提案流程**修改，不接受无记录的直接修改
- Claude Code 每次派发任务必须引用具体的 Spec 章节和版本

### 2.2 提案文档（Proposal）

一个变更从提出到交付的完整记录。存放在 `docs/proposals/`。

每个提案是一个独立目录，包含该变更的所有相关文档。

### 2.3 文档间关系

```
提案（变更的生命周期）
  ├── proposal.md    ← 变更是什么、为什么做
  ├── plan.md        ← 怎么一步步实现
  ├── spec-delta.md  ← 会对 Spec 造成什么影响
  └── decisions/     ← 开发中的设计决策

         ↓ 提案批准后合并

Spec（系统的当前定义）
  ├── PRD.md
  ├── ARCHITECTURE.md
  └── UI-DESIGN.md

         ↓ 全部完成后归档

Archive（历史记录）
  └── 已完成的提案
```

## 3. 提案生命周期

每个需求变更通过提案走完 **提出 → 评审 → 开发 → 验收 → 归档** 五个阶段。

### 3.1 提出阶段

**触发条件**：新功能、架构调整、API 变更、安全策略变更、技术栈变更等任何会影响 Spec 的变更。

**不需要提案**：Bug 修复、代码重构（不改行为）、样式微调、依赖版本升级（无破坏性）。

**判断标准**：这次变更会让 `docs/spec/` 中的某个文档需要更新吗？是 → 提案；否 → 直接做。

**步骤**：

1. 创建提案目录 `docs/proposals/<编号>-<描述>/`
2. 编写 `proposal.md`（提案描述）
3. 如涉及 Spec 变更，编写 `spec-delta.md`（变更预览）
4. 编写 `plan.md`（实现计划）

**此阶段禁止修改 Spec。**

### 3.2 评审阶段

**步骤**：

1. PM（化刚）审阅提案文档
2. 讨论、修改提案内容（可多轮）
3. PM 做出决策：**批准 / 拒绝 / 待补充**

**决策结果**：
- **批准** → 进入开发阶段
- **拒绝** → 提案标记为已拒绝，保留在 `proposals/` 作为历史记录
- **待补充** → 补充信息后重新评审

### 3.3 开发阶段

**步骤**：

1. 将 `spec-delta.md` 中的变更**合并**进 `docs/spec/` 对应文档
2. Spec 文档版本号递增，更新日期
3. 我（小梦旋）按 `plan.md` 拆分 Task，通过 Claude Code 派发
4. Claude Code 参照 Spec 编写代码
5. `plan.md` 中的 Task checkbox 随完成状态实时勾选
6. 开发中如需偏离 Spec，记录在 `decisions/` 并讨论

**规则**：
- 提案批准后 `proposal.md` 锁定，不再修改
- `plan.md` 是动态文档，可以随实际情况调整
- 偏离 Spec 的实现不能私自执行，必须先讨论再决定
- Claude Code 派发任务时引用格式：
  ```
  参照 docs/spec/ARCHITECTURE.md v4 第 3 节实现 SQL 解析。
  按 docs/proposals/001-mvp-initial/plan.md Task 3.1 验收标准。
  ```

### 3.4 验收阶段

**步骤**：

1. 全部 Task 完成
2. 我做初步验收：对照 Spec 检查实现是否符合规格
3. PM 做最终验收：功能是否符合预期
4. 如有问题 → 返回开发阶段修复
5. 验收通过 → 进入归档阶段

### 3.5 归档阶段

**步骤**：

1. 编写 `summary.md`（归档总结：做了什么、实际 vs 计划、偏离记录、经验教训）
2. 同步 `decisions/` 中未合并的 Spec 变更
3. 整个提案目录移入 `docs/archive/`
4. `proposals/` 中删除该目录

**归档后**：
- 提案不再活跃，仅作历史参考
- Spec 中已包含该提案的所有变更
- 后续如有相关问题，可查阅归档记录了解决策背景

## 4. 提案目录结构

### 4.1 标准结构

```
proposals/<编号>-<描述>/
├── proposal.md       # 提案描述（提出阶段创建，评审通过后锁定）
├── plan.md           # 实现计划（提出阶段创建，开发中动态更新）
├── spec-delta.md     # Spec 变更预览（涉及 Spec 变更时创建）
├── decisions/        # 设计决策记录（开发中按需创建）
│   └── <编号>-<描述>.md
└── summary.md        # 归档总结（归档阶段创建）
```

### 4.2 编号规则

格式：`<三位数字>-<kebab-case 描述>`

```
001-mvp-initial
002-add-mongodb-aggregation
003-implement-dingtalk-notification
```

编号全局递增，不可复用。

### 4.3 文件模板

#### proposal.md

```markdown
# Proposal: <标题>

> 状态: 草稿 | 评审中 | 已批准 | 已拒绝
> 创建日期: YYYY-MM-DD
> 更新日期: YYYY-MM-DD
> 关联 Spec: spec/xxx.md

## 概述

一两段话描述这个提案要做什么、为什么做。

## 背景

为什么要做这件事？解决什么问题？

## 范围

- 包含什么
- 不包含什么

## Spec 变更

列出会影响哪些 Spec 文档、具体改什么。详见 `spec-delta.md`。
如无 Spec 变更，写"无"。

## 风险与依赖

- 技术风险
- 依赖条件

## 验收标准

什么算"做完了"？列出可验证的具体条件。
```

## 5. Spec 文档维护规范

### 5.1 版本号

每个 Spec 文件头部包含版本信息：

```markdown
> 版本: vX | 日期: YYYY-MM-DD | 状态: 当前有效
```

版本递增规则：

| 变更类型 | 操作 | 示例 |
|----------|------|------|
| 措辞/格式修正 | 不升版本，只更新日期 | v4 → v4（日期更新） |
| 新增章节/补充细节 | 次版本 +1 | v4 → v4.1 |
| 功能范围/架构调整 | 主版本 +1 | v4 → v5 |

### 5.2 修改原则

- Spec 只通过提案合并来修改
- 紧急修复可以先改 Spec 再补提案，但必须在 24 小时内补上
- 每次修改更新文件头日期

## 6. 角色与职责

| 角色 | 人员 | 职责 |
|------|------|------|
| PM | 化刚 | 评审提案、验收交付、做需求决策 |
| PM + QA | 化刚 | 发现 bug、提出改进 |
| 调度者 | 小梦旋 | 拆分 Task、派发 Claude Code、跟踪进度、汇报 |
| 开发者 | Claude Code | 参照 Spec 编写代码 |
