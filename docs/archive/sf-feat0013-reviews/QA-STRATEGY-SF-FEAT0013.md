## 一、测试范围

### 1.1 变更性质

本次需求**不是框架迁移**（shadcn/ui 迁移已完成 ~90%），而是**设计规范对齐和视觉优化**。核心变更：

| 变更类型 | 具体内容 | 测试影响 |
|---------|---------|---------|
| CSS 变量重写 | 全套色值 token 从 Slate 系切换为设计规范色值 | 🔴 全页面视觉回归 |

**⚠️ 不需要重写单元测试**——单元测试验证的是行为逻辑，不是视觉表现。但如果测试中硬编码了 Tailwind 类名做断言（如 `expect(el).toHaveClass('bg-blue-500')`），需要同步更新为设计规范的新类名。

### 2.2 E2E 测试（Playwright）

| 目录 | 文件数 | 行数 | 覆盖内容 |
|------|-------|------|---------|
| `e2e/` | 20 个 spec | ~5,596 | 各页面功能测试（含 mock API） |
| `e2e-real/` | 5 个 spec | ~1,953 | 真实后端集成测试 |
| `tests/e2e/` | 2 个 spec | ~548 | 补充 E2E 测试 |
| **合计** | **27 个 spec** | **~8,096** | — |

**E2E 测试受影响分析**：

| 影响类型 | 说明 | 涉及测试 |
|---------|------|---------|
| CSS 类名选择器 | `page.locator('.some-class')` 如果目标类名变更 | 需逐一排查 |
| 文本选择器 | `page.getByText('搜索...')` 等，如果 placeholder/文案变更 | 少量 |
| 角色选择器 | `page.getByRole('button', { name: '...' })`，aria-label 变更 | 少量 |
| 布局断言 | `page.locator('...').boundingBox()` 如果尺寸变更 | `theme-switch.spec.ts` 等 |
| 主题断言 | `toHaveAttribute('data-theme', 'dark')` | 逻辑不变 |
| 视觉断言 | `toHaveScreenshot()` 如果存在 | 🔴 直接受影响 |

**结论**：E2E 测试以功能验证为主，大部分**不受视觉变更影响**。但需要注意：
1. 如果 E2E 测试中有截图对比（`toHaveScreenshot`），色值变更会导致全部失败，需要重新生成基准截图
2. 部分选择器可能因组件重构而失效，需要逐一验证

## 九、评审意见汇总

### 9.1 ✅ 积极方面

1. **设计规范质量高**：苏晴的规范覆盖全面（设计系统 + 8 个页面 + 通用组件 + 动效 + 响应式 + 无障碍），给测试提供了明确的标准
2. **测试资产丰富**：现有 12 个单元测试 + 27 个 E2E spec，总计 ~11,900 行测试代码，覆盖面广
3. **shadcn/ui 稳定性好**：组件结构稳定，样式变更通过 CSS 变量控制，回归风险可控
4. **分批交付合理**：方远的 P0→P1→P2→P3 分批策略与测试策略天然匹配

### 9.2 ⚠️ 需关注事项

1. **CSS 变量全局影响**：色值变更是一把双刃剑，改好了全站受益，改错了全站出问题。P0 阶段必须一次性做对
2. **浅色主题是薄弱点**：当前浅色主题变量不完整，P0 补全后需要**专项验证**，不能只在深色主题下测试
3. **单元测试 CSS 类名断言**：`Layout.test.tsx` 等文件中可能有 `toHaveClass('bg-xxx')` 断言，色值变更后需要同步更新
4. **新增页面无测试覆盖**：脱敏规则、AI 配置页面目前没有专门测试，建议在 P2 阶段补充

### 9.3 📋 建议事项

|---|------|--------|-------|
| 1 | P0 完成后立即建立视觉回归基线截图（至少 Login + Dashboard + Query） | 🟡 中 | QA |
