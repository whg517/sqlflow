## 需求概述

新增查询结果数据对比功能，允许用户选择两次查询结果进行行列级差异对比。

## 技术方案

### 现有功能

| 功能 | 说明 |
|------|------|
| 查询结果表格 | `ResultTable.tsx` |
| 列筛选 | 已支持 |
| CSV 导出 | 已支持 |
| 查询历史 | `query_history` 表，存储元数据（**不存储查询结果**） |
| 异步导出 | `ExportAsyncService`，结果存文件 |

### 关键问题：数据来源

**query_history 不存储查询结果**，对比时需要数据来源：

| 方案 | 优点 | 缺点 | 评估 |
|------|------|------|------|
| A. 重新执行 SQL | 无需存储 | 数据可能已变化，不可靠 | ❌ 不推荐 |
| B. 新增快照存储 | 数据可靠 | 需要存储空间 | ✅ 推荐 |
| C. 前端缓存 | 实现简单 | 刷新丢失 | ❌ 不可靠 |

**推荐方案 B**：复用 SF-FEAT0038（共享链接）的 `shared_results` 表设计思路，新增 `query_snapshots` 表或直接复用 `shared_results`。

### 实施方案

#### 数据存储（方案 B）

```sql
CREATE TABLE IF NOT EXISTS query_snapshots (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    query_history_id INTEGER NOT NULL,
    columns_json    TEXT NOT NULL,    -- 列定义
    rows_json       TEXT NOT NULL,    -- 查询结果（限制行数）
    total_rows      INTEGER NOT NULL, -- 原始总行数
    created_at      DATETIME NOT NULL DEFAULT (datetime('now'))
);
```

#### 后端

| 改动 | 说明 |
|------|------|
| 快照存储 API | `POST /api/query/snapshot`（查询执行时可选保存快照） |
| 对比 API | `POST /api/query/compare`（传入两个快照 ID，返回 diff 结果） |
| Diff 算法 | 服务端行级 diff（按行号匹配），返回新增/删除/修改行 |

#### 前端

| 改动 | 说明 |
|------|------|
| DiffView 组件 | 并排/合并视图，差异高亮 |
| 历史选择器 | 从查询历史中选择两条记录对比 |
| Schema 校验 | 仅允许列数和列名一致的结果对比 |

#### 工时估算（上调后）

| 任务 | 工时 |
|------|------|
| 快照存储 + 对比 API（后端） | 3h |
| DiffView 组件 + 历史选择器（前端） | 4h |
| Diff 算法 + 行匹配 | 1h |
| 测试验证 | 1h |
| **合计** | **9h**（上调自 4h） |

## 验收标准

1. 查询执行时可选择保存结果快照
2. 从查询历史中选择两条记录发起对比
3. Diff 视图正确显示差异：新增行（绿色）、删除行（红色）、修改行（黄色）
4. 仅允许相同 schema 的结果对比（列数和列名一致）
5. 大结果集（>100 行）分页展示差异
6. 与 SF-FEAT0038（共享链接）无冲突
