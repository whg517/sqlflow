# SQLFlow 文档管理规范

> 基于 OpenSpec Spec-Driven Development (SDD) 理念
> 版本: v1 | 日期: 2026-05-02

---

## 核心理念

**Spec 是真相来源（Source of Truth）**，所有开发必须参照 Spec 进行。文档不是写完就放一边的附属品，而是驱动整个开发流程的合约。

## 目录结构

```
docs/
├── GUIDELINES.md           # 本文件，文档管理规范
├── spec/                    # 系统规格（真相来源）
│   ├── PRD.md              # 产品规格——系统要做什么
│   ├── ARCHITECTURE.md     # 技术架构——系统怎么构建
│   └── UI-DESIGN.md        # UI 规格——界面怎么呈现
├── proposals/               # 活跃提案（开发中的）
│   └── 001-mvp-initial/    # 提案目录（编号递增）
└── archive/                 # 已归档提案（已完成）
```

## 三阶段生命周期

### 阶段一：提案（Proposal）

**目标**：在动手之前，对齐"做什么、为什么做、怎么做"。

**触发时机**：新功能、架构调整、重大修复、技术债务清理等

**流程**：

1. **创建提案目录**：`proposals/<编号>-<简短描述>/`
2. **编写提案文件**：
   - `proposal.md` — 提案描述
   - `plan.md` — 实现计划
   - `spec-delta.md` — Spec 变更预览（如有）
3. **评审提案**：PM（化刚）审批

**关键规则**：
- 提案阶段**不修改** `spec/` 目录
- `spec-delta.md` 明确列出会对哪些 Spec 文档造成什么影响
- 提案可以反复讨论和修改，直到审批通过

#### proposal.md 模板

```markdown
# Proposal: <标题>

> 状态: 草稿 | 评审中 | 已批准 | 已拒绝
> 创建日期: YYYY-MM-DD
> 更新日期: YYYY-MM-DD
> 关联 Spec: spec/xxx.md

## 概述

一两段话描述这个提案要做什么、为什么做。

## 背景

为什么要做这件事？解决什么问题？

## 范围

- 包含什么
- 不包含什么

## Spec 变更

列出会影响哪些 Spec 文档、具体改什么。详见 `spec-delta.md`。

## 风险与依赖

- 技术风险
- 依赖条件

## 验收标准

什么算"做完了"？
```

#### plan.md 模板

```markdown
# Plan: <标题>

> 状态: 待启动 | 开发中 | 已完成
> 更新日期: YYYY-MM-DD

## Task 拆分

按 Phase 或模块拆分，每个 Task 含 checkbox。

### Phase X: <名称>（X 工作日）

#### Task X.1 <任务名>

- [ ] 子任务 1
- [ ] 子任务 2

**验收**：具体可验证的标准

## 依赖关系

Task 之间的先后依赖

## 工时估算

| 阶段 | 工时 |
|------|------|
| ...  | ...  |
```

#### spec-delta.md 模板

```markdown
# Spec 变更预览

> 提案: <编号>
> 状态: 待合并 | 已合并

## spec/PRD.md

### 新增内容

<描述>
### 修改内容

<旧内容> → <新内容>
### 删除内容

<描述>

## spec/ARCHITECTURE.md

（同上结构）

## spec/UI-DESIGN.md

（同上结构）
```

### 阶段二：开发（Implementation）

**目标**：按 Plan 执行，Spec 指导代码，代码不偏离 Spec。

**流程**：

1. **合并 Spec 变更**：提案批准后，将 `spec-delta.md` 内容合并进 `spec/` 目录
2. **派发开发任务**：我（小梦旋）参照 `plan.md` 通过 Claude Code 派发 Task
3. **更新 Task 状态**：完成的 Task 在 `plan.md` 中勾选
4. **记录设计决策**：开发中偏离 Spec 的地方记录下来

**关键规则**：
- Claude Code 每次派发任务时**必须引用** `spec/` 中的具体章节
- 如果实现过程中发现需要调整 Spec，**先讨论再改**，不能私自改 Spec
- `plan.md` 是动态文档，随开发进度实时更新
- 不需要修改 Spec 的小调整（如 bug 修复、代码重构），直接开发，不写提案

#### 开发中需要的额外文件

```
proposals/<编号>-<描述>/
├── proposal.md       # 已批准，锁定
├── plan.md           # Task 状态实时更新
├── spec-delta.md     # 已合并后标记"已合并"
└── decisions/         # 开发过程中的设计决策（按需创建）
    ├── 01-xxx.md     # 决策编号 + 简短描述
    └── 02-yyy.md
```

#### decision.md 模板

```markdown
# Decision: <描述>

> 日期: YYYY-MM-DD
> 背景: <为什么需要这个决策>

## 上下文

遇到了什么问题？

## 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| A    | ...  | ...  |
| B    | ...  | ...  |

## 决策

选择了什么，为什么。

## Spec 影响

是否需要同步更新 Spec？如需要，描述变更内容。
```

### 阶段三：归档（Archive）

**目标**：提案完成后归档，保持 `proposals/` 目录整洁。

**流程**：

1. **确认所有 Task 完成**：`plan.md` 全部勾选
2. **验收通过**：PM 确认功能符合预期
3. **同步剩余 Spec 变更**：开发中的 `decisions/` 如涉及 Spec，最终合并
4. **编写归档总结**：
5. **移动到 archive/**：整个提案目录移入

#### summary.md 模板

```markdown
# Summary: <标题>

> 提案编号: <编号>
> 开始日期: YYYY-MM-DD
> 完成日期: YYYY-MM-DD
> 状态: 已完成 | 已取消

## 完成内容

概述完成了什么。

## 实际 vs 计划

| 维度 | 计划 | 实际 |
|------|------|------|
| 工时 | Xd   | Xd   |

最终对 Spec 的所有变更列表。

## 偏离记录

```

## 提案编号规则

格式：`<三位编号>-<kebab-case 描述>`

```
001-mvp-initial
002-add-mongodb-aggregation
003-implement-dingtalk-notification
```

编号递增，不可复用。

## 什么需要提案

| 需要 | 不需要 |
|------|--------|
| 新功能 | Bug 修复 |
| 架构调整 | 代码重构（不改行为） |
| API 变更 | 配置调优 |
| 数据库 Schema 变更 | UI 样式微调 |
| 安全策略变更 | 文档错别字修正 |
| 技术栈变更 | 依赖版本升级（无破坏性） |

**判断标准**：如果这次变更会让 `spec/` 中的某个文档需要更新 → 需要提案。

## Spec 文档维护规范

### 文件头

每个 Spec 文档必须包含版本信息：

```markdown
> 版本: vX | 日期: YYYY-MM-DD | 状态: 当前有效
```

### 状态值

- `草稿` — 初稿，待评审
- `评审中` — 正在讨论
- `当前有效` — 当前系统规格
- `已废弃` — 被新版本替代

### 版本号

- 小修（措辞、格式）→ 不升版本，只更新日期
- 中修（新增章节、补充细节）→ 递增次版本（v4 → v4.1）
- 大修（功能范围变更、架构调整）→ 递增主版本（v4 → v5）

### 修改原则

- Spec 只通过**提案合并**来修改，不接受直接修改
- 紧急修复可以先改 Spec 再补提案，但必须在 24 小时内补上
- 每次修改必须在文件头更新日期
