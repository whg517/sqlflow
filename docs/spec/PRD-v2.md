## v2.0 目标

v2.0 聚焦三大方向，不新增后端模块，以**前端体验打磨**和**代码质量治理**为主：

1. **用户体验增强**：浅色主题、全局搜索增强、编辑器自动补全
2. **前端功能补齐**：查询结果列筛选、审计日志增强
3. **代码质量提升**：ESLint 警告清零、TypeScript 严格模式

**设计原则**：
- 每个需求必须可测试、可验收
- 不引入新后端模块，仅利用已有 API
- 不破坏已有功能，增量迭代

## 新增功能需求

### 1. 浅色主题切换

#### 背景

v1.0 的 CSS Design Token 体系（`index.css`）已完整定义了深色和浅色两套变量。`[data-theme='light']` 选择器下的浅色 token 值已就绪，但缺少用户侧切换入口和持久化机制。

#### 用户故事

> 作为用户，我希望在深色和浅色主题之间切换，以适应不同的工作环境和个人偏好，特别是在光线充足的环境下使用浅色主题减轻眼睛疲劳。

#### 功能描述

1. **切换入口**：顶栏头像下拉菜单中新增「浅色/深色主题」切换项，图标使用 `Sun` / `Moon`（Lucide）
2. **持久化**：主题偏好存储在 `localStorage`（key: `theme`），值为 `'dark'` 或 `'light'`
3. **DOM 切换**：在 `<html>` 或 `<body>` 元素上设置 `data-theme="light"` / `data-theme="dark"` 属性
4. **初始化**：应用启动时从 localStorage 读取偏好，未设置时默认深色主题
5. **即时生效**：切换无需刷新页面，CSS 变量切换实现主题变化
6. **CodeMirror 适配**：SQL 编辑器的 `sqlTheme`（`SqlEditor.tsx`）需响应主题变化，动态更新背景色、文字色、选中色等样式
7. **系统偏好**（可选）：支持 `prefers-color-scheme` 媒体查询，当 localStorage 无设置时跟随系统偏好

#### 涉及文件

- `web/src/components/Layout.tsx` — 添加主题切换菜单项
- `web/src/index.css` — 已有浅色 token，无需修改
- `web/src/pages/Query/components/SqlEditor.tsx` — 编辑器主题适配
- `web/src/pages/Query/components/MongoEditor.tsx` — 编辑器主题适配
- 新建 `web/src/hooks/useTheme.ts` — 主题管理 hook

#### 验收标准

| # | 标准 | 验证方式 |
|---|------|---------|
| 1 | 点击头像下拉中的主题切换项，页面在深色/浅色之间即时切换 | 手动操作 |
| 2 | 刷新页面后主题保持上次选择 | 刷新验证 |
| 3 | 首次使用（无 localStorage）默认为深色主题 | 清除 localStorage 验证 |
| 4 | 切换主题后 SQL 编辑器（CodeMirror）的背景色、文字色、选中色正确切换 | 视觉检查 |
| 5 | 切换主题后所有页面（Dashboard、查询、工单、审计、权限、设置）样式正确 | 逐页检查 |
| 6 | 切换主题后所有弹窗（Dialog、Sheet、AlertDialog）样式正确 | 触发各类弹窗检查 |
| 7 | 浅色主题下文字对比度满足 WCAG AA 标准（对比度 ≥ 4.5:1） | 使用浏览器开发者工具检查 |

### 2. SQL 编辑器自动补全

#### 背景

v1.0 的 SQL 编辑器使用 CodeMirror 6，已引入 `@codemirror/autocomplete` 依赖，但未实现基于后端元数据的表名/字段名补全。后端已提供 `GET /api/datasources/:id/tables` 接口返回库表列表。

#### 用户故事

> 作为开发人员，我希望在编写 SQL 时能自动补全表名和字段名，减少拼写错误，提高查询编写效率。

#### 功能描述

1. **补全触发**：
   - 输入时自动触发（after 2+ characters）
   - `Ctrl+Space` 手动触发
2. **补全数据源**：
   - **关键字**：SQL 保留字（SELECT, FROM, WHERE, JOIN, ORDER BY 等）
   - **表名**：基于当前选中的数据源 + 数据库，调用 `GET /api/datasources/:id/tables` 获取
   - **字段名**：当光标在 `表名.` 之后时，获取该表的字段列表
3. **补全 UI**：
   - CodeMirror 内置补全弹出框，显示补全项 + 类型标签（table / column / keyword）
   - 支持模糊匹配（输入 `sel` 匹配 `SELECT`）
   - 键盘上下选择 + Enter/Tab 确认
4. **数据缓存**：
   - 表名列表缓存在前端，切换数据源/库时重新加载
   - 字段列表按表名缓存，避免重复请求
   - 缓存有效期 5 分钟
5. **MongoDB 编辑器**：
   - 补全集合名（collection）和常用操作符（$match, $group, $project 等）

#### 涉及文件

- `web/src/pages/Query/components/SqlEditor.tsx` — 添加 `autocompletion` 扩展
- `web/src/pages/Query/components/MongoEditor.tsx` — 添加 JSON 操作符补全 + 集合名补全
- `web/src/api/query.ts` 或新建 `web/src/api/metadata.ts` — 元数据 API 封装
- 新建 `web/src/hooks/useSchemaCompletion.ts` — Schema 补全数据 hook
- `web/src/pages/Query/index.tsx` — 传递数据源/库名给编辑器

#### 验收标准

| # | 标准 | 验证方式 |
|---|------|---------|
| 1 | 在 SQL 编辑器中输入 `SEL`，自动弹出补全列表包含 `SELECT` | 输入验证 |
| 2 | 在 `FROM` 关键字后输入时，弹出当前库的表名列表 | 选择数据源+库后验证 |
| 3 | 在 `表名.` 后输入时，弹出该表的字段名列表 | 输入验证 |
| 4 | 补全列表支持模糊匹配，输入 `usr` 能匹配到 `users` 表 | 输入验证 |
| 5 | `Ctrl+Space` 手动触发补全弹窗 | 快捷键验证 |
| 6 | 切换数据源或数据库后，补全列表更新为新库的表名 | 切换后验证 |
| 7 | 网络异常时补全不阻塞编辑器使用（优雅降级） | 断网测试 |
| 8 | 表名数据有 5 分钟前端缓存，避免频繁请求 | Network 面板观察 |
| 9 | MongoDB 编辑器中输入 `$` 自动弹出操作符补全列表（$match, $group 等） | MongoDB 模式输入验证 |
| 10 | MongoDB collection 输入框支持集合名补全 | 选择 MongoDB 数据源后验证 |

### 3. 全局搜索增强（Cmd+K）

#### 背景

v1.0 的 `CommandPalette` 组件已实现 `Cmd+K` 快捷键和页面跳转功能，但仅支持硬编码的页面导航列表。UI-DESIGN.md §全局交互规范 要求搜索范围包含工单（按 SQL 内容）和审计日志（按 SQL/表名），但目前缺少后端搜索集成。

#### 用户故事

> 作为用户，我希望通过 `Cmd+K` 全局搜索快速定位到历史查询、工单和审计日志，而不仅仅是页面跳转，从而提高日常工作效率。

#### 功能描述

1. **搜索分组**：

   | 分组 | 数据源 | 搜索方式 |
   |------|--------|---------|
   | 快速跳转 | 页面路由 | 本地匹配（已有） |
   | 查询历史 | `GET /api/query/history?keyword=` | 后端模糊搜索 |
   | 变更工单 | `GET /api/tickets?keyword=` | 后端模糊搜索 |
   | 审计日志 | `GET /api/audit-logs?keyword=` | 后端模糊搜索 |

2. **搜索行为**：
   - 输入关键词后 300ms debounce 发起后端搜索
   - 每个分组最多显示 5 条结果
   - 结果显示：SQL 摘要（前 60 字符）+ 数据源名称 + 时间
   - 高亮匹配关键词
3. **键盘导航**：
   - 上下键选择、Enter 跳转/执行
   - Esc 关闭搜索面板
4. **空状态**：无输入时显示最近 5 条查询历史作为推荐
5. **跳转行为**：
   - 点击查询历史 → 打开新查询 Tab 并填入 SQL
   - 点击工单 → 打开工单详情抽屉
   - 点击审计日志 → 跳转审计页并高亮对应记录

#### 涉及文件

- `web/src/components/CommandPalette.tsx` — 扩展搜索逻辑
- `web/src/api/query.ts` — 复用 `fetchHistory` 并添加 keyword 参数
- `web/src/api/ticket.ts` — 工单搜索 API（如未封装则新建）
- `web/src/api/audit.ts` — 复用 `listAuditLogs` 并添加 keyword 参数

### 6. 代码质量提升

#### 6.1 ESLint 警告修复

##### 背景

当前前端代码有 **6 个 ESLint 问题**（4 errors + 2 warnings），涉及以下类别（需逐一排查）：
- `react-hooks/set-state-in-effect` — Effect 中同步调用 setState（4 处 error）
- `react-hooks/incompatible-library` — TanStack Table 不兼容 React Compiler memo（1 处 warning）
- 未使用的 eslint-disable 指令（1 处 warning）

> 注：代码库经 Sprint 1-2 开发后已修复大部分 ESLint 问题，当前仅剩上述 6 项。PRD 初稿中记录的 52 个警告为旧数据，已更新。

##### 用户故事

> 作为开发团队，我们希望消除所有 ESLint 警告，保持代码库整洁，避免潜在 bug，并为 TypeScript 严格模式铺平道路。

##### 功能描述

1. **清零目标**：将 6 个 ESLint 问题全部修复，`npm run lint` 零警告零错误
2. **修复原则**：
   - `set-state-in-effect` 错误：将 Effect 中的数据获取重构为事件回调或 `useSyncExternalStore`
   - `incompatible-library` 警告：TanStack Table 已知限制，可添加 eslint-disable 并附注释说明
   - 删除无用的 eslint-disable 注释
   - 不新增 `eslint-disable` 注释（除非有充分理由并添加注释说明）
3. **CI 集成**：修复后在 CI 流水线中添加 lint 检查步骤（如尚未配置）

##### 验收标准

| # | 标准 | 验证方式 |
|---|------|---------|
| 1 | `npm run lint` 输出 0 warnings, 0 errors | 命令执行 |

#### 6.2 TypeScript 严格模式

##### 背景

当前 `tsconfig.json` 可能未开启严格模式。开启严格模式可以在编译期捕获更多类型错误，提升代码健壮性。

##### 用户故事

> 作为开发团队，我们希望开启 TypeScript 严格模式，利用编译器在开发阶段发现类型错误，减少运行时 bug。

##### 功能描述

1. **开启配置**：在 `tsconfig.json` 中设置 `"strict": true`，启用以下检查：
   - `strictNullChecks` — null/undefined 检查
   - `strictFunctionTypes` — 函数类型严格检查
   - `noImplicitAny` — 禁止隐式 any
   - `noImplicitThis` — 禁止隐式 this
   - `strictBindCallApply` — bind/call/apply 严格检查
2. **修复策略**：
   - 逐步修复：先开启 strict，根据编译错误逐一修复
   - 优先修复核心模块（API 层、Store、查询页）
   - 允许临时添加 `// @ts-expect-error` 注释标记待修复项（需附 GitHub Issue 跟踪）
3. **类型增强**：
   - API 响应类型全部定义 interface，消除 `as` 类型断言
   - 组件 Props 使用显式 type 定义
   - 事件处理器使用正确的事件类型

##### 验收标准

| # | 标准 | 验证方式 |
|---|------|---------|
| 1 | `tsconfig.json` 中 `"strict": true` | 文件检查 |
| 2 | `tsc -b` 编译通过，0 errors | 命令执行 |
| 3 | `npm run build` 构建成功 | 命令执行 |
| 4 | 所有页面功能正常，无回归 | 手动测试所有页面 |
| 5 | 不存在无注释的 `@ts-ignore`（应使用 `@ts-expect-error` 并附原因） | grep 验证 |

## 不做的事（v2.0 排除）

以下功能明确不在 v2.0 范围内：

- ❌ **新后端模块** — v2.0 不新增后端功能模块，仅利用已有 API

- ❌ **数据库性能监控** — 不做慢查询分析、性能仪表盘
- ❌ **多级审批流** — 保持单级 DBA 审批
- ❌ **跨数据源 JOIN** — 单次查询只连一个数据源
- ❌ **钉钉 OAuth** — 保持用户名密码登录
- ❌ **移动端适配** — 仅桌面端
- ❌ **SQL 格式化** — CodeMirror 快捷键已支持基本格式化
- ❌ **审计日志全文搜索（FTS5）** — 保持现有模糊匹配，FTS5 列入 v2.1 规划
- ❌ **数据比对/同步** — 不做跨库数据对比或同步
